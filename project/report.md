## 2. Постановка задачи и контекст

**Предметная область:** Задача Text-to-SQL позволяет пользователям взаимодействовать с базами данных без знания языка запросов. Потенциальный пользователь — аналитик или менеджер, который формулирует вопрос на естественном языке и получает готовый SQL-код.

**Формулировка задачи:** 
- Вход: текстовый вопрос и схема базы данных (таблицы и колонки)
- Выход: корректный SQL-запрос, соответствующий схеме и семантике вопроса
- Подзадача: мультилейбл-классификация для предсказания структуры запроса (набор интентов)

**Целевые метрики:**
- **Hamming Loss** — средняя доля ошибочно предсказанных меток
- **Subset Accuracy (Exact Match)** — доля примеров с абсолютно верным набором интентов
- **F1 Macro / Micro** — баланс между точностью и полнотой для частых и редких классов

## 3. Данные

**Источники:**
- Датасет **Spider** (официальный репозиторий) — базовое обучение
- Датасет **BIRD** (HuggingFace: `Sudnya/bird-sql`) — дообучение на сложных кейсах

**Структура:**
- Формат: JSON-файлы с парами `question` — `gold_query`
- Сопутствующие данные: схемы БД в формате SQLite
- Типы данных: текст (вопросы, SQL), структурные метаданные (схемы таблиц)

**Предобработка и подготовка:**
1. Извлечение интентов из эталонного SQL с помощью регулярных выражений и парсинга ключевых конструкций (`JOIN`, `GROUP BY`, `COUNT()` и др.)
2. Преобразование схемы БД в текстовый формат: `table: [col(type)]`
3. Конкатенация вопроса и схемы: `question [SCHEMA] table: [col(type)]`
4. Разделение выборки: train/val/test в пропорции 60/15/25 с фиксированным seed=42

Все этапы предобработки реализованы в ноутбуке `notebooks/exp001_EDA_n_MultiLabelClassifier.ipynb`.

## 4. Модели и подходы

**Базовая модель (baseline):**
- Эмбеддинги: `sentence-transformers/all-MiniLM-L6-v2`
- Классификатор: `OneVsRestClassifier(LogisticRegression)`
- Результат: базовое понимание структуры, но низкая точность на редких и сложных комбинациях интентов

**Финальная модель:**
- Архитектура: **DistilBERT** (`distilbert-base-uncased`) с адаптацией под мультилейбл-классификацию
- Стратегия обучения:
  1. Fine-tuning на датасете Spider (5 эпох, lr=2e-5)
  2. Продолжение обучения на датасете BIRD (15 эпох, lr=1e-5) для улучшения обобщения на сложных запросах
- Порог классификации: 0.5 для бинаризации логитов

Эксперименты проведены в ноутбуке `notebooks/exp001_EDA_n_MultiLabelClassifier.ipynb`.

## 5. Экспериментальный протокол и результаты

**Протокол:**
- Фиксированный random seed = 42 для воспроизводимости
- Разделение данных: train/val/test (60/15/25)
- Метрики рассчитываются на отложенной тестовой выборке
- Сравнение проводится по одинаковому набору меток, полученных через `MultiLabelBinarizer`

**Сравнение моделей:**

| Модель | Описание | Hamming Loss ↓ | Subset Accuracy ↑ | F1 Macro ↑ | Комментарий |
|--------|----------|----------------|-------------------|------------|-------------|
| Baseline | LogisticRegression + MiniLM | 0.1205 | 0.2862 | 0.5888 | Базовый уровень, высокая ошибка на редких классах |
| DistilBERT (Final) | Fine-tuned on Spider + BIRD | **0.0397** | **0.6839** | **0.6887** | Финальная модель, значительный прирост по всем метрикам |

**Выбор финальной модели:**
Выбран DistilBERT, дообученный на объединённой выборке Spider+BIRD. Модель демонстрирует снижение ошибки более чем в 3 раза и обеспечивает точность полного совпадения интентов ~68%, что критично для корректной последующей сборки SQL-запроса.

## 6. Архитектура решения и сервис

**Пайплайн обработки запроса:**
```
Входной запрос 
    ↓
[Классификатор] → Предсказание набора интентов (WhereQ, JoinQ, AggregationQ, ...)
    ↓
[Генератор параметров] → Ollama/Qwen генерирует JSON с параметрами для каждого интента
    ↓
[Валидатор схемы] → Проверка соответствия колонок и типов из схемы БД
    ↓
[SQLAssembler] → Сборка итоговой SQL-строки
    ↓
Выход: корректный SQL-запрос
```

**API (FastAPI):**
- `GET /health` — проверка статуса сервиса и готовности моделей
- `POST /api/classify` — классификация интентов по вопросу и схеме
- `POST /api/generate` — полный пайплайн: классификация → генерация параметров → сборка SQL

**Технологический стек:**
- Язык: Python 3.10
- Фреймворки: FastAPI, Pydantic, PyTorch, HuggingFace Transformers
- Инфраструктура: Docker, Docker Compose
- LLM-инференс: Ollama с моделью `qwen3:1.7b`

## 8. Ограничения и направления развития

**Текущие ограничения:**
- Разметка интентов получена эвристически из SQL-текста, что может вносить шум в редкие классы (`HavingQ`, `NullCheckQ`)
- Поддерживается только английский язык входных вопросов
- Сгенерированный SQL проверяется только на синтаксическую корректность и соответствие схеме, без исполнения в реальной БД
- Зависимость от внешнего LLM (Ollama) добавляет задержку и требует локальных вычислительных ресурсов

**Направления развития:**
- Внедрение execution-based валидации: пробный `EXPLAIN` или запуск в sandbox SQLite для проверки корректности запроса
- Активное обучение: выбор сложных/неуверенных примеров для ручной разметки и дообучения
- Оптимизация инференса: квантизация модели (`bitsandbytes`), батчинг запросов, fallback на rule-based генератор при отказе LLM
- Расширение языковой поддержки: дообучение на мультиязычных данных или использование multilingual-эмбеддингов

2. **Сложный запрос с агрегацией и сортировкой:**
   ```json
   {

     "schema": {
       "employees": {"id": "INT", "dept_id": "INT", "salary": "FLOAT"},
       "departments": {"id": "INT", "name": "VARCHAR"}

     "base_table": "employees"
   }
   ```
