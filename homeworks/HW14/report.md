# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- **База знаний**: ~100 коротких статей из русской Википедии (населённые пункты, персоналии, понятия). Выбрана как учебный набор с разнообразной тематикой и лаконичными текстами.
- **Чанкинг**: разбиение по символам с `chunk_size=250`, `overlap=50`. Простой и воспроизводимый подход.
- **Эмбеддинги**: модель `all-MiniLM-L6-v2` (SentenceTransformers), нормализация L2, индекс `FAISS IndexFlatIP` (косинусное сходство).
- **Retrieval**: Hit@5 = 0.600, Recall@5 = 0.600, MRR@5 = 0.267. Половина запросов находит релевантный документ в топ-5.
- **Обновление базы**: добавлено 3 документа (Этика ИИ, Обучение с подкреплением, Трансформеры). Для 2 из 3 тестовых запросов состав выдачи изменился.
- **Mini-RAG**: генерация ответов через Qwen 0.5B. Модель склонна к повторам и галлюцинациям, но демонстрирует полный конвейер: запрос → retrieval → контекст → ответ.

## 3. База знаний и подготовка документов

- Предметная область: энциклопедические статьи (география, персоналии, наука).
- Источник: локальный JSON-файл со списком `{'url': ..., 'text': ...}`.
- Число исходных документов: ~100.
- Число чанков после разбиения: ~180–220 (зависит от длины статей).
- Способ чанкинга: последовательное разбиение по символам с перекрытием.
- Параметры: `chunk_size=250`, `overlap=50`.
- Комментарий: короткие вики-статьи хорошо подходят для демонстрации retrieval — каждый документ содержит компактный факт. Однако разнообразие тем повышает риск семантических коллизий (похожие формулировки в разных контекстах).

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- Embedding-модель: `sentence-transformers/all-MiniLM-L6-v2`.
- Метрика сходства: косинусное (через нормализацию L2 и Inner Product в FAISS).
- Индекс FAISS: `IndexFlatIP`.
- Default `top_k`: 5.
- Первые примеры retrieval: для запросов вроде "Что такое атеизм?" модель корректно находила статью "Сильный и слабый атеизм"; для географических запросов иногда возвращала другие населённые пункты со схожей структурой текста.

### 4.2. Контрольные запросы и оценка retrieval

- Контрольных запросов: 10.
- Релевантным считался чанк из статьи, соответствующей ожидаемому `title`.
- Метрики: `hit@k`, `recall@k`, `MRR@k`.
- Hit@5: 0.600
- Recall@5: 0.600
- MRR@5: 0.267
- Самые простые запросы: фактологические с уникальными ключевыми словами ("Потсдамская конференция", "Подорожник Корнута красная книга").
- Самые проблемные: общие формулировки ("село Украина", "год основания") — много ложных срабатываний из-за похожей структуры статей.

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

- Сравниваемый параметр: `chunk_size`.
- Варианты: 300 и 500 символов.
- Результат: Hit@5 = 0.400 (300) vs 0.200 (500). Меньший чанк лучше сохраняет локальный контекст для коротких запросов.
- Выбранный вариант: `chunk_size=250` (компромисс между целостностью фрагмента и точностью).
- Обновление базы: добавлено 3 новых документа по темам ИИ.
- Переиндексация: полное пересоздание эмбеддингов и индекса FAISS.
- Изменения в retrieval: для запросов "этика искусственного интеллекта" и "архитектура трансформеров" новые документы появились в топ-3; для "обучение с подкреплением" выдача не изменилась (возможно, из-за семантической близости к другим техническим статьям).

## 7. Анализ

Выбранная база знаний оказалась удачной для учебного пайплайна: короткие фактологические статьи позволяют быстро проверить работу retrieval. Чанкинг по символам с небольшим перекрытием сохранил смысловые единицы, но для более длинных текстов потребовалось бы разбиение по предложениям или абзацам. Формальная оценка (`hit@5=0.6`) дала объективную метрику: половина запросов находит нужный документ, что приемлемо для прототипа. Эксперимент с `chunk_size` показал, что для коротких вики-статей меньший размер чанка даёт лучшую точность. Обновление базы знаний продемонстрировало, что добавление тематически близких документов действительно меняет выдачу — но только если запрос достаточно специфичен. Ошибки mini-RAG в основном связаны с ограничениями малой модели (0.5B): она не всегда умеет выделять главное из контекста и склонна к шаблонным повторам. Основные ограничения решения: отсутствие reranking, эвристический чанкинг, малая мощность генеративной модели, нет обработки многошаговых запросов.

## 8. Итоговый вывод

Базовым конвейером для такой задачи оставил бы: `all-MiniLM-L6-v2` + `FAISS IndexFlatIP` + `chunk_size=250` + `top_k=5`. Главное, что я понял про качество retrieval: даже хорошая модель эмбеддингов не компенсирует плохо сформулированный запрос или неудачный чанкинг. Связь retrieval и качества mini-RAG прямая: если в контексте нет нужного фрагмента, даже идеальная генеративная модель не даст правильный ответ. Поэтому оптимизация retrieval — приоритет перед улучшением генерации.

## 9. Приложение (опционально)

- Дополнительные метрики: `MRR@5 = 0.267` указывает, что релевантный документ часто находится не на первой позиции.
- Анализ ошибок: запросы с общими словами ("село", "год", "основан") дают много ложных срабатываний из-за лексического перекрытия статей.
- Рекомендация: для улучшения качества можно добавить reranking (например, cross-encoder) или использовать более крупные модели эмбеддингов (`bge-m3`, `multilingual-e5`).
